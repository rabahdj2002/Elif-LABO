# Corrigibility

**Principle:** the architecture must be allowed to be wrong, and to be corrected by reality.

**ELIF application:**
- Components must be removable, not just addable
- The procedure can shrink as well as grow
- Compression to a smaller architecture is a valid outcome, not a failure
- The repo itself is corrigible: doctrine, procedure, and component list all subject to evidence-driven revision

**Test of real corrigibility:** can the project ship the outcome where ELIF reduces to ≤3 components, and accept it as a legitimate answer? If not, the corrigibility commitment is cosmetic.

See Rule 2 in `doctrine/02_locked_rules.md`.
