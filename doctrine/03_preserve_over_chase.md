# Preserve-over-chase

**Principle:** if a behavior fails to emerge from the current architecture, restructure or isolate before brute-forcing repeated attempts.

**ELIF application:**
- If a procedure step produces weak or awkward outputs, that is benchmark data, not a signal to silently modify the procedure mid-execution
- If a component fails to engage across a case, log it; do not invent reasons it "would have" worked
- If a benchmark exposes a fundamental ambiguity, log it as benchmark-blocking and revise — not iterate blindly

**Inherited from:** OpenStudyGo production friction — `feedback_preserve_over_chase_doctrine`. Validated under operational conditions in formation generation waves.
